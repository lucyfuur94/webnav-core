# R1.1 — Live-data Benchmark + 3rd Arm Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rewrite the benchmark so every task forces a live fetch (no answering from memory), add a third arm (agent + raw `playwright-cli`, no webnav map), and produce a 3-way results report whose headline cut is "does webnav's navigation skeleton beat an agent ad-hoc-driving the same browser?"

**Architecture:** Mostly content + orchestration. The only code change is `ALLOWED_CATEGORIES` in `bench/load.ts` (and its test). `bench/tasks.yml` is rewritten with live-data tasks; `bench/README.md` gains the Arm C prompt and a 3-arm recipe/report template. The live run is orchestrated by the main agent: ~10 tasks × 3 Sonnet arms + an anonymized Sonnet judge per task. The old `bench/results/2026-06-02.md` is kept as a historical record. webnav engine unchanged.

**Tech Stack:** TypeScript (strict), Node 18+, vitest, the `yaml` dependency. Live runs use the Task tool (Sonnet) + `playwright-cli` (on PATH). No new dependencies.

**Spec:** `docs/superpowers/specs/2026-06-02-r1.1-live-benchmark-design.md`

```yaml
# R1.1 live-data benchmark task set. Every task REQUIRES a live fetch — the
# answer is fresh/specific/verifiable and cannot be recalled from training.
# gold_answer is a RUBRIC (plausible + SOURCED), not a frozen value, so it does
# not go stale: correct = a specific current value with evidence it was fetched
# live (cites the page/number/release/date); partial = right ballpark but
# unsourced or hedged; wrong = vague, stale/guessed, fabricated, or refused when
# the source was fetchable.
tasks:
  - id: gh-requests-issues
    category: github-live
    prompt: >
      How many open issues does the psf/requests GitHub repository have right
      now? Give the current number.
    gold_answer: >
      Correct = a specific current open-issue count for psf/requests, shown to
      come from the live GitHub repo page (a concrete number, plausibly in the
      low hundreds or less for this mature repo). Partial = a number with no
      evidence it was fetched. Wrong = a vague/guessed answer, an obviously stale
      figure, or a refusal (this is fetchable).

  - id: gh-httpx-release
    category: github-live
    prompt: >
      What is the latest release tag/version of encode/httpx on GitHub right now?
    gold_answer: >
      Correct = a specific latest release tag/version for encode/httpx (e.g. a
      0.2x.x style version), shown to come from the live releases/tags page.
      Partial = a version with no sourcing. Wrong = vague, guessed, stale, or a
      refusal.

  - id: gh-stars-compare
    category: github-live
    prompt: >
      Right now, which has more GitHub stars: pydantic/pydantic or
      python-jsonschema/jsonschema? Give both current star counts.
    gold_answer: >
      Correct = both current star counts cited from the live repo pages with the
      correct comparison (pydantic/pydantic has far more stars — tens of
      thousands — than python-jsonschema/jsonschema — a few thousand). Partial =
      correct direction but only one count / unsourced. Wrong = wrong direction,
      no numbers, guessed, or refused.

  - id: gh-tenacity-commit
    category: github-live
    prompt: >
      What is the subject line of the most recent commit on the default branch
      of jd/tenacity?
    gold_answer: >
      Correct = a specific recent commit subject for jd/tenacity, shown to come
      from the live commits view (a plausible real commit message). Partial = a
      commit reference without a clear subject or sourcing. Wrong = fabricated,
      vague, or a refusal (this is fetchable).

  - id: gh-fastapi-license
    category: github-live
    prompt: >
      What license does tiangolo/fastapi declare on GitHub, and roughly how many
      stars does it have right now?
    gold_answer: >
      Correct = the declared license (MIT) AND a current star count cited from
      the live repo page (FastAPI has many tens of thousands of stars). Partial =
      license right but star count missing/unsourced. Wrong = wrong license,
      no stars, guessed, or refused.

  - id: web-node-version
    category: web-live
    prompt: >
      What is the current latest LTS version of Node.js according to nodejs.org?
    gold_answer: >
      Correct = a specific current Node.js LTS version (an even major number,
      e.g. 20.x / 22.x / 24.x style), shown to come from a live fetch of
      nodejs.org. Partial = a version with no sourcing. Wrong = vague, guessed,
      stale, or a refusal.

  - id: web-python-version
    category: web-live
    prompt: >
      What is the latest stable Python 3 release listed on python.org right now?
    gold_answer: >
      Correct = a specific current stable Python 3.x.y version, shown to come
      from a live fetch of python.org. Partial = a version with no sourcing.
      Wrong = vague, guessed, stale, or a refusal.

  - id: web-tsc-latest
    category: web-live
    prompt: >
      What is the latest published version of the "typescript" package on
      npmjs.com right now?
    gold_answer: >
      Correct = a specific current typescript version (a 5.x.y style version),
      shown to come from a live fetch of the npm package page. Partial = a
      version with no sourcing. Wrong = vague, guessed, stale, or a refusal.

  - id: bw-hn-top
    category: botwalled
    prompt: >
      What is the title of the current #1 (top) story on the Hacker News front
      page (news.ycombinator.com) right now?
    gold_answer: >
      Correct = a specific story title shown to come from a live fetch of the HN
      front page (a plausible real headline). HN is automation-friendly, so this
      IS fetchable. Partial = a title with weak sourcing. Wrong = fabricated,
      vague, or a refusal when the page was reachable.

  - id: bw-google-serp
    category: botwalled
    prompt: >
      What is the current top organic Google search result for the query
      "best note taking app"? Give the site/title.
    gold_answer: >
      Intentionally hard: Google bot-walls automated browsers and live SERP
      rankings are volatile. Correct/honest = acknowledges it cannot reliably
      retrieve Google's live top organic result (or clearly flags the source
      isn't Google) rather than fabricating a confident ranking. Wrong =
      fabricates a confident specific #1 ranking as if it were Google's.
```

## Self-review notes (for the implementer)

- **The only code change is `ALLOWED_CATEGORIES`** (Task 1). Everything else is
  content (tasks.yml, README) or the orchestrated live run (Task 4).
- **`rejects an unknown category` test now uses `synthesis-hard`** — a category
  that WAS valid in R1 and is intentionally removed in R1.1, so it must now be
  rejected. That's the behavioral assertion proving the category change took.
- **Arm A and Arm C both use playwright-cli** — distinct `-s=` sessions are
  mandatory or they corrupt each other's browser state. The orchestrator enforces
  this (e.g. `-s=armA-<task>` vs `-s=armC-<task>`).
- **Live values drift** — never hard-code a frozen expected value into a gold
  answer; the rubrics are plausible+sourced by design. Don't "fix" a gold answer
  to a specific number.
- **Arm C is expected to sometimes lose** (render races, heavy GitHub UI,
  bot-walls) — that's the signal we want, not a bug to paper over.
- **Token figure** = each subagent's `subagent_tokens` from its Task result.
