# Navigation benchmark (saucedemo) — 2026-06-13

Run of `docs/superpowers/specs/2026-06-03-navigation-benchmark-design.md`
(2026-06-12 addendum: saucedemo walk flows). Tasks: `bench/tasks-nav.yml` (4).
Arms on **Sonnet** (held constant across arms — the CLAUDE.md benchmark
exception): **A** = webnav CLI only · **C** = raw playwright-cli only ·
**B** = WebSearch/WebFetch (run once, as the honest can't-do demonstration).

## Headline

- **Quality: tied.** A 4/4 correct · C 4/4 correct. saucedemo is simple enough
  that a competent agent gets there either way.
- **Reliability is where the arms separate: A 3 clean / 1 recovered · C 1 clean
  / 3 recovered.** saucedemo drops the login session on some direct-URL
  navigations; raw-browser runs fell into re-login loops in 3 of 4 tasks. The
  one A run that hit the same trap was the run where the agent chose raw-style
  driving.
- **Median tool calls: A 22.5 · C 22.5** (toolbox parity when the agent drives
  manually — see finding 1). Tokens: A ~38k · C ~35k median (spawn-floor caveat
  ~18.6k applies; tool_uses is the primary cost signal per the design).
- **B: structurally can't.** It honestly reported it cannot log in, then
  derived numerically-correct totals from third-party test repos + the known 8%
  tax rate — without ever reaching the page. Rubric: wrong (no live evidence);
  honesty + ingenuity noted.

## The finding that matters most

**1. Agents default to manual driving — the walk wins when actually used.**
In 3 of 4 A runs the Sonnet agent ignored `walk` and drove with `use
snapshot/click/type` (the familiar loop), making arm A ≈ arm C with different
branding. The one task where the agent leaned on the walk
(`nav-detail-description`) was the cheapest, cleanest run of the entire
benchmark: **9 tool calls, 68 s** (vs 22.5 median elsewhere). The bottleneck is
walk **ergonomics/discoverability**, not capability.

**2. Orchestrator-measured walk cost (the autopilot, used as designed):**
login → checkout-overview in **4 agent-visible CLI calls** — `walk` (pauses at
the add-to-cart fork with a snapshot), one `use click`, `walk-resume --ref` —
with the checkout form **auto-filled from stored creds** (12 playwright calls
under the hood, zero agent reasoning about navigation). The same journey cost
the manual arms 16–22 reasoned actions, each ingesting a snapshot.

**3. UX wart (follow-up filed):** after firing a gate affordance there is no
"gate satisfied, continue" answer — `walk-resume` accepts only `--ref`/
`--classify`, so the agent must re-answer with a ref even though the gate is
done. On an icon-only element (the cart badge) that's a genuine stumble.
→ **R5.1: add a bare-continue resume** (re-attempt the paused step; the
addressable jump branch then handles icon-only links).

## Caveats / deviations from the spec

- 4 tasks, not ~8 (walk-shaped saucedemo flows; the GitHub set was retired by
  the 2026-06-12 addendum).
- Quality scored by the orchestrator against the objective rubrics (exact
  dollar values), not an anonymized judge — the rubric values leave no room for
  taste; noted as a deviation.
- Arm B run once (demonstration) rather than per-task: it has no browser, so
  per-task repetition only re-proves the same structural inability.
- Traces are self-reported by the arms; tool_uses/tokens come from the Task
  results (floor caveat above).
- Single run per cell; saucedemo's session-drop behavior is real but its
  frequency across runs is not characterized.

## Verdict against the thesis

On a simple site the navigation skeleton does NOT beat manual driving on
*quality* — both arms get there. It wins on **reliability** (the map already
knows the cart is an addressable jump; raw driving rediscovers the session trap
every time) and on **cost when the walk is actually used** (4 calls vs ~20).
The actionable conclusion is product, not benchmark: make the walk the obvious
first move (R5.1 continue-resume, better help copy, possibly a single
`webnav go <goal>` verb), then re-measure.
