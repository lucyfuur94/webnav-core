# R1 results — 2026-06-02

Model: sonnet (both arms + judge). Tasks: 12 from bench/tasks.yml.
NOTE: live run — numbers vary run-to-run; this is one sample.

## Headline

- **Answer quality: webnav 3 / baseline 1 / tie 8** (of 12)
- **Median agent-token delta (baseline − webnav): −6,076 tokens** — NEGATIVE, i.e.
  webnav used *more* agent tokens on this task mix, not fewer (mean 28.5k vs 20.4k).

**The honest takeaway (and it's not the naive thesis):** webnav did NOT save agent
tokens here, and on quality it tied most tasks while winning the ones that need
*fresh / verifiable / bot-walled* data. The reason webnav costs more: the baseline
agent answered most well-known questions **from prior knowledge with zero tool
calls** (~18.8k-token floor), while webnav *always navigates* (real CLI + browser),
adding 5–22k tokens. So on general-knowledge questions webnav's navigation is pure
overhead. webnav's real, demonstrated edge is **answer quality on data an LLM can't
just recall**: live maintenance signals, and content behind bot-walls/paywalls that
plain WebFetch can't read.

Reading it:
- **github-discovery** — webnav matched the baseline on quality (3 ties) and won
  once (`gh-compare-maintenance`: webnav compared concrete recent signals for BOTH
  libs; the baseline leaned on popularity → judged partial). But webnav cost
  ~10.8k more tokens median — its multi-page GitHub navigation (up to 25 tool
  calls on the comparison task) is expensive vs the baseline's targeted WebSearch.
- **open-web-info** — 3 ties + webnav's only LOSS (`ow-langchain`: webnav honestly
  punted because Marginalia/Wiby don't index a LangChain definition page; the
  baseline answered from prior knowledge). This is the documented open-web-search
  cap, not a bug — webnav does not evade the engines that bot-wall it.
- **synthesis-hard** — both tied (both correct on CAP, both partial on monorepo).
  webnav has no edge on synthesis (expected) and costs ~10.5k more tokens for
  navigating sources an LLM didn't need.
- **botwalled** — **webnav swept 2–0**, the clearest win. On `bw-paywall` webnav's
  real browser read the NYT *free homepage* and returned a real current lead
  headline; the baseline's WebFetch was blocked and (correctly) punted. On
  `bw-google` webnav honestly explained it can't see Google's SERP; the baseline
  **fabricated** a confident "Zapier is #1" ranking → judged wrong. This is exactly
  the predicted edge: a real browser sees rendered/free content WebFetch can't, and
  webnav reports honest uncertainty instead of hallucinating.

## What this means for webnav

- **The token-savings framing of the thesis does NOT hold for general info-seeking.**
  When the answer is something a strong model already knows, the cheapest agent just
  answers — webnav's navigation is overhead. Token savings would require tasks where
  the agent CANNOT answer without navigating AND webnav navigates more cheaply than
  the agent driving a browser itself (not exercised by this mix).
- **webnav's demonstrated value is QUALITY on non-recallable data:** fresh maintenance
  signals (won the head-to-head comparison) and bot-walled/paywalled/rendered content
  (swept the botwalled category, including catching a baseline hallucination). It also
  never fabricated — it punts honestly when its engines can't reach a source.
- **Actionable next experiments:** to test the token thesis fairly, add tasks that
  REQUIRE multi-step navigation to non-addressable state (where a plain agent would
  burn many tokens reading huge snapshots) and measure there. To strengthen the
  quality story, add more fresh-data / bot-walled tasks where recall can't substitute
  for retrieval.

## Caveats

- One live sample; re-running varies. Judge is an LLM (fallible) — spot-check close
  calls (esp. the `partial` calls on gh-compare-maintenance and the synthesis tasks).
- Open-web/botwalled tasks are capped by automation-friendly engines (webnav does not
  evade bot-walls); the baseline's WebSearch/built-in knowledge may legitimately win
  there — and did on `ow-langchain`.
- Sample size 12; treat as directional evidence, not a significance test.
- `bw-paywalled-fact`: webnav was judged *correct* for returning a real headline
  from the NYT FREE homepage, which over-delivers vs the task's rubric (which only
  expected an honest "can't retrieve paywalled content" punt). Defensible — a real
  retrieval beats a punt — but if a re-run punts instead, that "correct" won't
  reproduce. A genuinely paywalled-only fact would harden this task.
- Token counts are each subagent's total `subagent_tokens` for the task, which mixes
  reasoning + tool I/O; it's the realistic "what the arm cost" figure, not isolated
  navigation cost.
