# R1.1 results — 2026-06-03 (3-arm, live-data, on the re-grounded CLI)

Model: sonnet (all three arms + judge). Tasks: 10 from bench/tasks.yml (live-data).
Arms: **A = webnav CLI** (now with `read`/`eval` + data-driven `recall`),
**B = baseline** (Claude WebSearch + WebFetch), **C = raw playwright-cli** (no webnav map).
Anonymized 3-way judge, answer order varied per task.
NOTE: live run — numbers vary run-to-run; this is one sample. This run uses the
**re-grounded CLI** (generic verbs + read/eval + playwright-style help), unlike the
original `bench/results/2026-06-02.md` which ran on the old GitHub-coupled surface.

## Headline

- **Answer quality (correct, of 10): webnav 9 · baseline 7 · raw-browser 9**
- **Median agent-tokens: webnav 21.7k · baseline 19.5k · raw-browser 21.7k**
- **KEY CUT (A vs C): webnav matched-or-beat raw-browser on quality in 10/10 tasks**
  (tied 10/10; neither lost to the other), at near-identical token cost.

**The takeaway — and it's the opposite of the first run's "webnav thrashes":** with
`read`/`eval` and data-driven `recall`, webnav is now a **top performer** — tied for
the best quality (9/10, alongside raw-browser) and never thrashed. The earlier
"how many open issues" task that made webnav flail in R1 it now answers cleanly in
a couple of calls. The honest cost picture: all three arms cluster ~19–22k tokens
because every task forces a real fetch; the baseline is ~2k cheaper because for a
few values its WebSearch/API hit returns the answer in 2 calls, while webnav and
raw-browser open + read a page. webnav's edge is **quality on the cases search/API
get subtly wrong or can't reach** (see below).

Where the arms diverged (8/10 tasks all three tied as correct):
- **gh-requests-issues — webnav & raw-browser correct, baseline PARTIAL.** Both
  page-reading arms returned **145** (the human-meaningful Issues-tab count). The
  baseline returned **229** from GitHub's API `open_issues_count`, which conflates
  issues + PRs — the subtly-wrong answer. Reading the actual page beat the API here.
- **web-tsc-latest — baseline correct, webnav & raw-browser PARTIAL.** npmjs.com
  bot-walls automated browsers (Cloudflare 403). webnav **honestly punted** ("I do
  not evade walls"); raw-browser also struggled. The baseline's WebSearch surfaced
  6.0.3 from its index. This is the documented open-web/bot-wall cap — the baseline
  legitimately wins, and webnav fails *honestly* rather than fabricating.
- **botwalled (bw-hn-top, bw-google-serp) — webnav & raw-browser correct, baseline
  PARTIAL (2/2).** HN: all three got the live top story. Google SERP: webnav and
  raw-browser both **honestly reported they can't get Google's ranking** (bot-wall /
  uses Marginalia); the baseline gave a specific result (Zapier) with a caveat —
  judged partial because the source isn't actually Google. Crucially, **no arm
  fabricated** this run (in the original R1 the baseline fabricated a confident
  Google #1 — fixed expectations here).

## A vs C focus (the thesis cut)

webnav (A) vs raw-browser (C) — *does webnav's map/primitives beat an agent
ad-hoc-driving the same browser?* On QUALITY they **tied 10/10** (same verdict on
every task) at near-identical median tokens (21.7k vs 21.7k). So on this live-data
set, webnav's navigation skeleton did **not** out-quality or out-cheap raw browsing
— BUT the crucial change from R1 is that webnav **no longer loses to it / thrashes**:
`read`/`eval` give the agent the same clean single-page access raw-browser has, with
a more legible interface. Where webnav *should* pull ahead of raw-browser is
multi-page non-addressable navigation (search→filter→drill-in) — not exercised by
these single-page lookups. Honest read: on single-value live lookups, webnav ≈
raw-browser ≈ baseline on quality; webnav's real wins were the two cases page-reading
beat the API (issue count) and honest-failure beat fabrication (bot-walled).

## Caveats

- One live sample; re-running varies. Judge is an LLM (fallible) — the `partial`
  calls (issue-count API, tsc bot-wall, Google SERP) are the close ones.
- Token figure = each subagent's total `subagent_tokens`, which is **dominated by a
  ~18–19k fixed spawn floor** (measured earlier: a no-op subagent ≈ 18.6k). So the
  ~2k spread between arms understates/overstates little; the real navigation-cost
  difference is small at single-page scale and largely hidden by the floor. A
  cleaner token metric (tool-result bytes ingested) is noted as future work.
- web-tsc + Google: webnav fails by DESIGN (no bot-wall evasion); the baseline's
  search index wins there. That's the honest ecosystem boundary, not a webnav bug.
- Single-page tasks don't exercise webnav's multi-step-navigation thesis; that needs
  a follow-up task set (search→filter→drill-in to non-addressable state).
